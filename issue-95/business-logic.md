### 操作流程

```
管理員進入「角色設定」頁
        ↓
  查看角色清單（可依角色名稱關鍵字篩選）
        ↓
  ┌────────────────────────────────┐
  │ 新增角色                        │
  │ → 開啟 Modal                   │
  │ → 填入名稱、勾選成員、          │
  │   AI 夥伴、知識庫、功能權限     │
  │ → 儲存                         │
  └────────────────────────────────┘
        ↓
  ┌──────────────────┐   ┌──────────────────────────────┐
  │ 修改角色          │   │ 刪除角色                       │
  │ → 開啟編輯 Modal  │   │ → 確認對話框                   │
  │ → 修改後儲存      │   │ → 清除關聯資料（user_roles、    │
  └──────────────────┘   │   role_ai_partners、role_kb_map）│
                         │ → 刪除角色主記錄                 │
                         └──────────────────────────────┘
```

### 功能權限（4 項，對應 schema roles 欄位）

依 Prototype `f_role.html` 呈現，UI 顯示 4 項功能權限 checkbox：

| UI 標籤      | Schema 欄位            | 說明                        |
|-------------|------------------------|-----------------------------|
| AI 夥伴      | `can_access_ai`        | 可使用 AI 夥伴進行分析        |
| 帳號管理     | `can_manage_accounts`  | 可新增、修改、刪除使用者帳號  |
| 角色管理     | `can_manage_roles`     | 可新增、修改、刪除角色與權限  |
| AI 夥伴管理  | `can_edit_ai`          | 可調整 AI 夥伴設定與綁定知識庫 |

> ⚠️ Schema 另有 `can_use_kb`（查閱知識庫）與 `can_manage_kb`（管理知識庫）兩個 flag，但 Prototype 未呈現為獨立 checkbox——知識庫存取改以「可存取知識庫」多選清單（`role_kb_map`）控制。SD 需與 PM 確認是否需保留或移除這兩個欄位。

### 角色刪除規則

- 刪除前需二次確認
- 刪除時需同步清除：`user_roles`、`role_ai_partners`、`role_kb_map` 的關聯資料
- 是否允許刪除仍有使用者掛載的角色：Prototype 未明示，SD 需確認（建議：禁止刪除並提示改為先解除成員）

### 存取控制

- 僅具備 `can_manage_roles = TRUE` 的使用者可進入角色管理頁面

## 資料模型示意（選填）

**⚠️ ── AI 填寫開始，請逐行審查 ──**

```
users ──< user_roles >── roles ──< role_ai_partners >── ai_partners
                           │
                           └──< role_kb_map >── knowledge_bases
```

| 實體              | 用途                        |
|-------------------|-----------------------------|
| `roles`           | 角色主表（名稱 + 6 項 permission flags） |
| `user_roles`      | 使用者 ↔ 角色（多對多）      |
| `role_ai_partners`| 角色 ↔ AI 夥伴（多對多）     |
| `role_kb_map`     | 角色 ↔ 知識庫（多對多）      |

> 四張表均已定義於 `schema.md`，無需新增資料表。

## SD 注意事項（選填）

**⚠️ ── AI 填寫開始，請逐行審查 ──**

1. **無需新增資料表**：`roles`、`user_roles`、`role_ai_partners`、`role_kb_map` 皆已存在於 schema.md。
2. **Schema vs Prototype 權限欄位差異**：Schema 有 6 個 permission flags（含 `can_use_kb`、`can_manage_kb`），Prototype 僅呈現 4 項；SD 需與 PM 確認是否調整 schema，再決定是否需要 Alembic migration 移除多餘欄位。
3. **角色名稱唯一性**：`roles.name` 已有 UNIQUE constraint，API 層需將 DB 唯一鍵衝突轉換為 422 Unprocessable Entity 回傳。
4. **成員批次指派**：Prototype 在新增/編輯 Modal 中同時管理成員（`user_roles`），儲存時需一次性 diff 並批次更新 `user_roles`，避免逐筆操作。
5. **關聯清除順序**：刪除角色前先清除 `user_roles`、`role_ai_partners`、`role_kb_map`，再刪除 `roles` 主記錄（或使用 DB CASCADE）。
6. **Permission Guard**：角色管理所有 API 須驗證 `can_manage_roles`，回傳 403 Forbidden。
7. **Prototype 參考**：`p1-design/Prototype/v75/f_role.html`
