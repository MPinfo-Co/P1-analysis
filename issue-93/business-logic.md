# 業務邏輯分析：使用者功能

## 需求說明

**⚠️ ── AI 填寫開始，請逐行審查 ──**

> Epic #109 body 為空；以下需求說明由 SA 根據 `p1-design/Spec/f_user/` 現有規格文件推導。

管理系統使用者帳號，提供具備帳號管理權限（`can_manage_accounts`）的角色執行帳號的新增、查詢、修改與刪除，以及角色指派。

### 驗收條件

- 可依角色與關鍵字篩選帳號清單
- 可新增帳號（姓名、Email、密碼、角色），Email 不可重複，密碼最少 8 字元，至少指派一個角色
- 可修改帳號（姓名、Email、角色），密碼修改為獨立功能不在本範疇
- 可刪除帳號，且不可刪除自己（避免管理員鎖定）
- 無 `can_manage_accounts` 權限者，無法存取本功能

## 商業邏輯（選填）

**⚠️ ── AI 填寫開始，請逐行審查 ──**

### 權限前置條件

所有操作（查詢、新增、修改、刪除）均需：
1. JWT 有效（已登入）
2. 操作者角色具 `can_manage_accounts = true`

### 新增帳號流程

```
填入姓名 / Email / 密碼 / 角色（至少一個）→ 點擊「儲存」
        ↓
後端驗證欄位
        ├─ Email 格式錯誤 → 400
        ├─ 密碼 < 8 字元 → 400
        ├─ role_ids 為空 → 400
        └─ Email 已存在 → 409（Email 欄位顯示「此 Email 已被使用」）
        ↓ 驗證通過
bcrypt hash 密碼 → 寫入 users（is_active = true）
逐筆寫入 user_roles
        ↓
201：關閉 Modal，重新整理清單
```

### 修改帳號流程

```
帶入現有資料（姓名 / Email / 已指派角色）→ 修改後儲存
        ↓
後端驗證（僅驗證有傳入的欄位）
        ├─ Email 與其他使用者重複 → 409
        ├─ role_ids 為空陣列 → 400
        └─ 使用者不存在 → 404
        ↓ 驗證通過
更新 users 欄位
若有傳入 role_ids：刪除現有 user_roles → 重新寫入（整組取代）
        ↓
200：關閉 Modal，重新整理清單
```

### 刪除帳號流程

```
點擊「刪除」→ 確認 Dialog（「確定要刪除此帳號嗎？無法復原。」）
        ↓ 確認
後端防呆檢查
        ├─ 操作者 = 目標 user → 400（不可刪除自己）
        └─ 使用者不存在 → 404
        ↓ 通過
刪除 user_roles（該 user 的所有角色關聯）
刪除 users 紀錄
        ↓
204：重新整理清單
```

### 篩選與搜尋規則

| 參數 | 行為 |
|---|---|
| role_id 未傳 | 回傳所有帳號 |
| role_id 傳入 | 篩選具該角色的帳號 |
| keyword 未傳 | 不做關鍵字過濾 |
| keyword 傳入 | 對 name / email 做 ILIKE 模糊搜尋（不分大小寫）|

結果依 `users.id` 升冪排序。

## 資料模型示意（選填）

**⚠️ ── AI 填寫開始，請逐行審查 ──**

涉及 `users`、`roles`、`user_roles` 三張既有資料表，均已存在 schema.md，無需新增。

```
users ──< user_roles >── roles
```

| 實體 | 關聯 | 說明 |
|---|---|---|
| users | 1 | 一位使用者 |
| user_roles | N | 可指派多個角色 |
| roles | 1 | 每個角色對應一筆 roles 紀錄 |

**外鍵注意**：`security_events.assignee_user_id` 為 NULLABLE FK → users。刪除使用者時，若該使用者曾被指派為事件處理人，需確認 DB 層 ON DELETE 行為（建議設為 SET NULL，避免違反 FK 約束）。

## SD 注意事項（選填）

**⚠️ ── AI 填寫開始，請逐行審查 ──**

1. **密碼安全**：密碼以 bcrypt hash 後儲存，API Request / Response 均不得出現明文密碼或 `password_hash` 欄位。
2. **密碼修改**：修改帳號不包含密碼欄位，密碼修改為獨立功能，本 Epic 不實作。
3. **角色取代策略**：修改帳號時若傳入 `role_ids`，採「先刪後插」整組取代，非增量更新。
4. **自我刪除防呆**：DELETE 端點需比對 JWT 中的 `user_id` 與目標 `id`，相同則回 400，防止管理員意外刪除自己。
5. **Email 唯一性**：`users.email` 已有 DB 層 UNIQUE 約束，後端同步做應用層檢查，提供語義明確的 409。
6. **FK 連動**：`security_events.assignee_user_id` FK，刪除使用者前需確認 migration 已設定 `ON DELETE SET NULL`，否則會觸發 FK 違反錯誤。
